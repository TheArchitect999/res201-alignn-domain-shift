## 3. Split Protocol and Run-Local Subset Construction

### Literature-grounded context

- Original JARVIS benchmark split identities were inherited before family filtering. `[CITE: JARVIS 2020 dataset/repository paper]` (split provenance only — do not overclaim benchmark formalism beyond what the repo supports).

### Implementation details

- Split identities from `provided:manifests/dft_3d_formation_energy_peratom_splits.csv` were preserved before family filtering.
- For each family, `pool = train + val`; `test` remains fixed and is never resampled.
- Run-local datasets are built by `scripts/shared/Prepare_Week1_Finetuning_Dataset.py`.
- For each family, each N, and each seed:
  - sample N structures from the family `pool.csv` using `np.random.default_rng(seed).permutation(...)[:N]`
  - allocate validation rows with `n_val = max(5, int(round(0.1 * N)))`; if `n_val >= N`, fall back to `max(1, N // 5)`
  - place validation rows first, then training rows, then the unchanged family test set
- `keep_data_order=True` plus explicit `n_train`, `n_val`, and `n_test` enforce the row order inside `id_prop.csv`.

### Experimental setup

- Fine-tuning was run at N = 10, 50, 100, 200, 500, 1000 for both families with five random seeds each.
- From-scratch reuses the same run-local dataset roots but only for N = 50 and N = 500.
- Family test sizes are fixed: 1484 for oxide, 242 for nitride.
- Manuscript tables should cite TAB_METHODS_DATASET_SPLITS_v1 for counts.

### Evidence anchors

- `data_shared/oxide/summaries/summary.json`
- `data_shared/nitride/summaries/summary.json`
- `scripts/shared/Prepare_Week1_Finetuning_Dataset.py`

### Implementation details

- Zero-shot evaluation is implemented in `scripts/shared/Evaluate_ALIGNN_Zero_Shot.py`.

- Do not call the checkpoint "oxide-pretrained". Use "pretrained formation-energy ALIGNN model".

## 5. Fine-Tuning Protocol

### Literature-grounded context

- "Fine-tuning initialized from the pretrained formation-energy ALIGNN checkpoint." `[CITE: ALIGNN foundational paper]`

### Implementation details

- Fine-tuning uses pretrained checkpoint `jv_formation_energy_peratom_alignn/checkpoint_300.pt` and its accompanying config.
- Implementation: `scripts/shared/Fine_Tune_Last_Two_ALIGNN_Layers.py`.
- All parameters are frozen first; only `gcn_layers.3` and `fc` are unfrozen.
- The overall model remains in `eval()` mode; `model.gcn_layers[3]` and `model.fc` are switched to training mode explicitly. Invariant checks ensure no unexpected parameters become trainable.
- Optimization: `AdamW`, `OneCycleLR`, `torch.nn.L1Loss()`.
  - Note: inherited config JSON files still contain `criterion: "mse"` from the pretrained checkpoint's original training config. The custom training script overrides this and uses L1Loss throughout.
- Best checkpoint selected by lowest validation L1.
- Outputs per-split prediction CSVs and `summary.json` with `best_epoch` and `test_mae_eV_per_atom`.

### Experimental setup

- Fine-tuning is run at N = 10, 50, 100, 200, 500, 1000 for both families.
- Each condition uses five random seeds.
- Total Set 1 fine-tuning runs: 60 (30 per family).
- Each run records `best_epoch` and `test_mae_eV_per_atom` in `summary.json`. The per-N `mean_best_epoch` reported in tables is the mean over five seeds. Methods defines this quantity so Results can interpret it; Methods does not interpret it.
- Results are summarized as mean ± SD across seeds.

### Evidence anchors

- `scripts/shared/Fine_Tune_Last_Two_ALIGNN_Layers.py`
- `Results_Hyperparameter_Set_1/README.md`
- `configs/Hyperparameter_Set_1/week2_finetune/oxide_week2_N500_seed0.finetune_last2.json`

## 9. Embedding-Analysis Protocol

**Main-text summary (manuscript-scale):**

Structure embeddings were extracted from the frozen pretrained ALIGNN model (checkpoint `jv_formation_energy_peratom_alignn/checkpoint_300.pt`) without retraining or modifying any prior result folders. The primary embedding layer for main-text analysis is `last_alignn_pool` — the pooled node tensor returned by the last ALIGNN convolutional block. Two additional layers (`pre_head` and `last_gcn_pool`) were extracted as robustness controls; in the current outputs they are numerical near-duplicates of each other and are reserved for appendix use, not treated as independent probes.

Three named evaluation subsets were constructed:

| subset | contents |
|---|---|
| `fixed_test_set` | all oxide test structures plus all nitride test structures |
| `balanced_pool_set` | all nitride pool structures plus an equal-size random oxide pool sample (seed 42) |
| `oxide_reference_pool` | all oxide pool structures |

Family separation was quantified in raw 256-dimensional embedding space using silhouette score, Davies-Bouldin index, k-nearest-neighbor family purity (k = 15), and logistic-regression AUC (5-fold cross-validation, StandardScaler). Raw-space statistics are the inferential layer; PCA, t-SNE, and UMAP projections are descriptive support only.

Nitride distance-versus-error analysis used: oxide centroid Euclidean distance, mean Euclidean distance to the 5 nearest oxide-reference embeddings, and supplemental Ledoit-Wolf Mahalanobis distance (after covariance screening). Hard and easy nitride groups are the top and bottom 20% of fixed-test nitrides by absolute zero-shot error.

Statistical defaults: 1000 bootstraps (family separation), 5000 bootstraps and 10 000 permutations (distance-error), within-statistic Benjamini-Hochberg FDR adjustment.

**Visualization parameters and overlay policy details** are in `embedding_methods_appendix_notes_v1.md`.

### Methods / Results boundary guardrails

- Methods may define `mean_best_epoch` as a recorded protocol quantity and explain checkpoint selection.
- Methods may describe the distance-versus-error protocol as procedural.
- Methods must NOT interpret low-N nitride runs as inert or effectively zero-shot. That is a Results/Discussion claim.
- Methods must NOT describe MAE trends or adaptation behavior.
- Methods must stay neutral about oxide versus nitride outcomes.
