## 1. Oxide Standalone — Methods

### 2.1 Dataset and prediction target

Crystal structures and their formation-energy labels were drawn from the JARVIS materials repository, specifically the `dft_3d_2021` release. `[CITE: JARVIS 2020 dataset/repository paper]` The supervised target throughout the oxide arm was the per-atom formation energy `formation_energy_peratom`, reported in eV/atom. A machine-readable summary of the oxide family composition is stored alongside the repository at `data_shared/oxide/summaries/summary.json`, and the filtering script used to construct it is `scripts/dataset/build_res201_family_datasets.py`.

### 2.2 Oxide family definition

We define an oxide as any structure whose composition contains oxygen. Structures that contain both oxygen and nitrogen (oxynitrides) are retained in the oxide arm; 499 such structures are present after filtering. Family assignment is performed by `scripts/dataset/res201_stage2_lib.py` through the `classify_family(...)` routine with `keep_oxynitrides_in_oxide=True`. For completeness when both arms are referenced in the same sentence, the complementary family used in related work is defined as structures containing nitrogen but no oxygen; that arm is not analyzed in this report.

### 2.3 Split protocol and run-local subset construction

We inherit the original JARVIS benchmark split identities (`provided:manifests/dft_3d_formation_energy_peratom_splits.csv`) before applying family filtering, so that each oxide structure keeps the train, validation, or test assignment it carries in the underlying release. `[CITE: JARVIS 2020 dataset/repository paper]` After filtering, the oxide family contains 14 991 structures, partitioned into 11 960 training, 1 547 validation, and 1 484 test structures; the pool (training plus validation) therefore holds 13 507 structures. These counts and their source are reported in Table 1 (TAB_METHODS_DATASET_SPLITS).

Run-local training subsets were constructed by `scripts/shared/Prepare_Week1_Finetuning_Dataset.py`. For each sample size N and each random seed, N structures were drawn from the oxide pool using `np.random.default_rng(seed).permutation(...)[:N]`. Within each subset, the number of validation examples was set to `n_val = max(5, round(0.1 × N))`, with training receiving the remainder; if `n_val` would otherwise equal or exceed N, the allocation falls back to `max(1, N // 5)`. The fixed oxide test set (n = 1 484) is appended unchanged to every run-local dataset. Row ordering inside each `id_prop.csv` is enforced with `keep_data_order=True` together with explicit `n_train`, `n_val`, and `n_test` arguments so that loader code sees a deterministic training, validation, and test sequence.

### 2.4 Zero-shot evaluation

We evaluated the pretrained formation-energy ALIGNN model on the fixed oxide test set without any task-specific training. `[CITE: ALIGNN foundational paper]` The evaluation is implemented in `scripts/shared/Evaluate_ALIGNN_Zero_Shot.py`, which loads the POSCAR files listed in `data_shared/oxide/manifests/test.csv` and obtains per-structure predictions through `alignn.pretrained.get_prediction(model_name="jv_formation_energy_peratom_alignn", atoms=atoms)`. Test MAE is computed with `sklearn.metrics.mean_absolute_error`. Per-structure predictions are written to `Results_Before_Correction/oxide/zero_shot/predictions.csv`, and the summary value is recorded in `reports/zero_shot/zero_shot_summary.csv`. Zero-shot evaluation is performed once per family and is not seed-varied.

### 2.5 Fine-tuning and from-scratch training protocols

Fine-tuning is implemented in `scripts/shared/Fine_Tune_Last_Two_ALIGNN_Layers.py` and operates on the pretrained checkpoint `jv_formation_energy_peratom_alignn/checkpoint_300.pt` together with its accompanying configuration. We freeze every parameter of the loaded model, then selectively re-enable training for the final gated GCN block (`gcn_layers.3`) and the scalar output head (`fc`). The overall model remains in `eval()` mode; only the two unfrozen modules are switched to training mode, and runtime invariant checks confirm that no additional parameters become trainable. Optimization uses `AdamW` with a `OneCycleLR` schedule and an L1 training objective (`torch.nn.L1Loss`); for transparency, we note that the inherited config JSON retains a `criterion: "mse"` entry from the original pretraining configuration, and the custom training script overrides this field and trains with L1Loss throughout. The best checkpoint per run is selected by minimum validation L1, and each run writes a `summary.json` that records `best_epoch` and `test_mae_eV_per_atom`. Fine-tuning was conducted at N ∈ {10, 50, 100, 200, 500, 1 000} with five random seeds per condition, for a total of 30 oxide fine-tuning runs.

From-scratch training, implemented in `scripts/shared/Train_ALIGNN_From_Scratch.py`, reuses exactly the same run-local dataset roots and model configuration as fine-tuning but instantiates a fresh ALIGNN model (`ALIGNN(ALIGNNConfig(**cfg.model.model_dump()))`) with no pretrained weights and all parameters trainable. The optimizer, learning-rate schedule, and L1 objective match the fine-tuning setup, and best-checkpoint selection again uses minimum validation L1. From-scratch baselines exist only at N = 50 and N = 500 for the oxide arm; each of the two conditions uses five random seeds, giving 10 oxide from-scratch runs in total. No from-scratch comparison exists at N = 10, 100, 200, or 1 000, and the Results and Discussion sections do not attempt transfer-benefit inferences outside the two sample sizes where both protocols are available.

### 2.6 Hyperparameter setting, model specification, and evaluation metric

All trained runs in the oxide arm use Hyperparameter Set 1 — the canonical namespace pinned by the project brief — with `epochs = 50`, `batch_size = 16`, and `learning_rate = 1 × 10⁻⁴`. The graph-construction settings carried by the shared configuration template are `neighbor_strategy = "k-nearest"`, `cutoff = 8.0` Å, `cutoff_extra = 3.0` Å, and `max_neighbors = 12`, with `use_canonize`, `compute_line_graph`, and `use_lmdb` all enabled. The underlying architecture is the ALIGNN formation-energy model: four ALIGNN convolutional layers combined with four gated GCN layers, a hidden size of 256, and a scalar output head. `[CITE: ALIGNN foundational paper]` No additional architecture table is introduced.

The primary reported metric is test-set mean absolute error (MAE) in eV/atom. Checkpoint selection for all trained protocols uses the lowest validation L1. Multi-seed results are summarized as the mean ± standard deviation across the five random seeds. The zero-shot value is a single evaluation on the fixed oxide test set and is reported without a dispersion estimate. The full experimental scope for the oxide arm — two zero-shot evaluations, 30 fine-tuning runs, and 10 from-scratch runs — is summarized in Table 2 (TAB_METHODS_EXPERIMENT_SCOPE).

### 2.7 Embedding-analysis bridge

To complement the error-based evaluation, we extracted structure embeddings from the frozen pretrained ALIGNN model (checkpoint `jv_formation_energy_peratom_alignn/checkpoint_300.pt`) without retraining and without modifying any fine-tuning or from-scratch outputs. The main-text embedding layer is `last_alignn_pool`, the pooled node tensor returned by the last ALIGNN convolutional block. Two additional layers (`pre_head` and `last_gcn_pool`) were extracted as robustness controls; in the current outputs they are numerical near-duplicates of each other and are treated as matched appendix-facing layers rather than as independent probes. Full visualization parameters, the full family-separation metric set, and the nitride distance-versus-error protocol are described in the combined paper and in `embedding_methods_appendix_notes_v1.md`; the oxide report uses this extraction only to confirm that the same pretrained representation space supports the family-separation bridge figure discussed in Results. Embedding analysis is used as an interpretive protocol, not as a proof of mechanism.

### 2.7 Frozen-embedding analysis protocol

Structure embeddings were extracted from the frozen pretrained ALIGNN model (checkpoint `jv_formation_energy_peratom_alignn/checkpoint_300.pt`) without retraining and without modifying any fine-tuning or from-scratch outputs. Extraction touches only inference code and therefore does not alter any previously reported training artefact.

The primary embedding layer for all main-text analysis is `last_alignn_pool`, the pooled node tensor returned by the last ALIGNN convolutional block. Two additional layers were extracted as robustness controls: `pre_head`, the pooled output of `model.readout` immediately before `model.fc`, and `last_gcn_pool`, the pooled node tensor from the last gated GCN block (`gcn_layers[3]`). In the current outputs, `pre_head` and `last_gcn_pool` are numerical near-duplicates of each other. We therefore treat them as a matched pair of appendix-facing supporting layers and do not present them as independent probes of the representation.

Three fixed evaluation subsets were constructed for the embedding analyses. The `fixed_test_set` is the union of all oxide test structures and all nitride test structures and is used for family-separation scoring and for nitride-side error metadata. The `balanced_pool_set` contains every nitride pool structure together with an equal-size random sample from the oxide pool (drawn with seed 42) and is used for balanced visualization and for bootstrap-resampled family-separation estimates. The `oxide_reference_pool` is defined as the set of all oxide pool structures; it provides the reference manifold against which nitride embedding distances are measured. Throughout this report, "oxide-reference region" is used only to describe positions in this manifold.

Family separation is quantified directly in the raw 256-dimensional embedding space. For each fixed subset we compute the silhouette score, the Davies-Bouldin index, the family purity of the k-nearest-neighbor graph at k = 15, and the AUC of a logistic-regression family classifier evaluated under five-fold cross-validation with `StandardScaler` preprocessing. These raw-space statistics are the inferential layer of the embedding analysis; PCA, t-SNE, and UMAP projections are used only as descriptive visual support, with primary settings `perplexity = 30` for t-SNE and `n_neighbors = 30`, `min_dist = 0.1` for UMAP, both fitted on standardized inputs without PCA pre-reduction; PCA itself is fitted on the `balanced_pool_set` after standardization and used to project the other subsets into that basis. Parameter sensitivity runs and overlay-fit policies are described in `embedding_methods_appendix_notes_v1.md`.

For the distance-versus-error analysis, each nitride test structure is characterized by three embedding-space distances to the oxide-reference manifold: the Euclidean distance to the oxide centroid, the mean Euclidean distance to its five nearest oxide-reference neighbours, and a supplemental Ledoit-Wolf-regularized Mahalanobis distance evaluated only when covariance-screening diagnostics pass. These distances are paired with per-structure absolute zero-shot error. To characterize the distribution of prediction difficulty relative to the embedding geometry, we define hard nitrides as the top 20 % of fixed-test nitrides by absolute zero-shot error and easy nitrides as the bottom 20 %. The hard/easy assignment is a protocol-level partition; interpretation of what the resulting distance differences imply for adaptation belongs to Results and Discussion.

Statistical defaults for the embedding analyses are: 1 000 bootstrap resamples for family-separation estimates, 5 000 bootstrap resamples and 10 000 permutations for distance-error tests, and within-statistic Benjamini-Hochberg false-discovery-rate adjustment. Throughout, the embedding workflow is treated as an interpretive protocol that produces evidence for later discussion, not as a mechanism-proving instrument in its own right.

## 3. Combined Paper — Methods

### 2.1 Study design, dataset, and prediction target

We study a single pretrained formation-energy ALIGNN model applied to two family-specific evaluation arms that share every procedural choice other than the identity of the retained structures. `[CITE: ALIGNN foundational paper]` The oxide arm serves as the in-distribution control and the nitride arm as the cross-family test; the comparison between them is a study-design choice made at the level of the project rather than a literature claim. Crystal structures and their supervised targets were drawn from the JARVIS materials repository, specifically the `dft_3d_2021` release. `[CITE: JARVIS 2020 dataset/repository paper]` The supervised target in both arms is the per-atom formation energy `formation_energy_peratom`, reported in eV/atom. Family-level summaries are stored at `data_shared/oxide/summaries/summary.json` and `data_shared/nitride/summaries/summary.json`; the script that builds both family datasets is `scripts/dataset/build_res201_family_datasets.py`.

### 2.2 Family definitions and filtering

Family assignment is element-presence based and is performed by `scripts/dataset/res201_stage2_lib.py` through `classify_family(...)` with `keep_oxynitrides_in_oxide=True`. The oxide arm contains every structure whose composition includes oxygen, and it retains 499 oxynitrides (structures containing both O and N). The nitride arm contains every structure whose composition includes nitrogen but not oxygen, and all oxynitrides are excluded from this arm by construction (zero retained). This filtering choice is asymmetric by design: it keeps the oxide arm fully populated with every oxygen-containing structure while guaranteeing that the nitride arm is free of mixed-anion cases.

### 2.3 Split inheritance and run-local subset construction

Original JARVIS benchmark split identities (`provided:manifests/dft_3d_formation_energy_peratom_splits.csv`) were preserved before family filtering, so each retained structure carries its existing train, validation, or test assignment into the family arm. `[CITE: JARVIS 2020 dataset/repository paper]` After filtering, the oxide family contains 14 991 structures (11 960 train / 1 547 validation / 1 484 test, pool = 13 507) and the nitride family contains 2 288 structures (1 837 train / 209 validation / 242 test, pool = 2 046). For each family, the test set remains fixed and is never resampled; the pool is the reservoir from which run-local subsets are drawn. These counts are reported in Table 1 (TAB_METHODS_DATASET_SPLITS).

Run-local datasets were built by `scripts/shared/Prepare_Week1_Finetuning_Dataset.py`. For each family, each sample size N, and each random seed, N structures were drawn from the family pool using `np.random.default_rng(seed).permutation(...)[:N]`. Validation allocation is `n_val = max(5, round(0.1 × N))`, with training receiving the remainder; if `n_val` would otherwise equal or exceed N, the allocation falls back to `max(1, N // 5)`. The fixed family test set is appended unchanged to every run-local dataset, and row ordering inside each `id_prop.csv` is enforced with `keep_data_order=True` together with explicit `n_train`, `n_val`, and `n_test` arguments to preserve a deterministic training, validation, and test sequence at load time.

### 2.4 Zero-shot baseline

Zero-shot evaluation (`scripts/shared/Evaluate_ALIGNN_Zero_Shot.py`) applies the pretrained formation-energy ALIGNN model to each family's fixed test set without any task-specific training. `[CITE: ALIGNN foundational paper]` The script loads the POSCAR files listed in each family's `test.csv` manifest and obtains per-structure predictions through `alignn.pretrained.get_prediction(model_name="jv_formation_energy_peratom_alignn", atoms=atoms)`; test MAE is computed with `sklearn.metrics.mean_absolute_error`. Per-structure outputs are written to `Results_Before_Correction/<family>/zero_shot/predictions.csv`, and the canonical summary is recorded in `reports/zero_shot/zero_shot_summary.csv`. Zero-shot evaluation is performed once per family on the fixed test set and is not seed-varied.

### 2.5 Partial fine-tuning protocol
